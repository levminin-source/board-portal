# Площадка Совета директоров — MVP

Веб-платформа для организации работы Совета директоров: заседания и повестка,
заочное голосование, согласование дат, материалы и протоколы, организационные
поручения функциональным директорам, уведомления и журнал действий.

Статус: **MVP / proof-of-concept** для демонстрации и последующей передачи
в ДЦТиИТ для доработки и переноса во внутренний информационный контур.

---

## 1. Быстрый запуск (нужен только Python 3.11+)

```bash
cd backend
pip install -r requirements.txt
python -m app.seed              # один раз: создаёт БД и демо-пользователей
uvicorn app.main:app --reload
```

Откройте в браузере: **http://127.0.0.1:8000**

Демо-доступы (пароль у всех: `demo123` — сменить перед реальным использованием):

| Логин       | Роль                     |
|-------------|--------------------------|
| `secretary` | Корпоративный секретарь  |
| `kuznetsov` | Председатель СД          |
| `minina`, `orlov`, `safonova` | Члены СД |
| `fin_dir`   | Финансовый директор (исполнитель поручений) |
| `it_dir`    | Директор по ИТ (исполнитель поручений)      |

Документация API (Swagger) генерируется автоматически: **http://127.0.0.1:8000/docs**

## 2. Архитектура и обоснование стека

```
board-portal/
├── backend/
│   ├── requirements.txt
│   └── app/
│       ├── db.py         # подключение к БД (SQLite → PostgreSQL заменой строки)
│       ├── models.py     # схема данных (SQLAlchemy)
│       ├── security.py   # bcrypt, JWT, блокировка при переборе паролей, роли
│       ├── main.py       # все маршруты REST API + раздача фронтенда
│       └── seed.py       # первичное наполнение демо-данными
├── frontend/
│   ├── index.html        # точка входа (React без этапа сборки)
│   └── app.jsx           # интерфейс: рабочий стол, заседания, поручения, архив, журнал
└── README.md
```

**Бэкенд — Python / FastAPI + SQLAlchemy.**
Выбран потому, что: (а) минимум кода для рабочего REST API с автодокументацией
(Swagger «из коробки» — удобно для приёмки ДЦТиИТ); (б) Python — наиболее
распространённая компетенция в штатных ИТ-командах; (в) SQLAlchemy позволяет
перейти с SQLite на корпоративный PostgreSQL заменой одной строки подключения.

**БД — SQLite для MVP.** Нулевая настройка, один файл `board.db`.
Для продакшна — PostgreSQL (см. раздел 4).

**Фронтенд — React без этапа сборки** (JSX компилируется в браузере).
Осознанный компромисс для MVP: проект запускается одной командой без Node.js
и npm, что критично при руководителе проекта без опыта разработки. Для
продакшна ДЦТиИТ переводит фронтенд на сборку Vite + TypeScript (структура
компонентов уже разделена и переносится почти механически) и размещает
React локально вместо CDN (в закрытом контуре внешние CDN недоступны).

**Роли и права** (проверяются на сервере, не только в интерфейсе):

| Возможность | Секретарь | Член СД | Функц. директор |
|---|---|---|---|
| Создание заседаний, материалов, статусы | ✔ | — | — |
| Просмотр заседаний, материалов, архива | ✔ | ✔ | — |
| Голосование, согласование дат, комментарии | — | ✔ | — |
| Создание поручений, приёмка исполнения | ✔ | — | — |
| Исполнение поручений (свои), отчёты | — | ✔ (если назначено) | ✔ |
| Журнал действий | ✔ | — | — |

**Проектные решения, согласованные с заказчиком:**
- голосование открытое, промежуточные итоги видны участникам до закрытия
  (п. 7.7 Положения о Совете директоров);
- голос подаётся однократно и не может быть изменён (сервер отклоняет повтор);
- кворум — настраиваемый параметр заседания (по умолчанию 3 из 4 —
  простое большинство);
- система фиксирует голоса; квалификацию результата («принято/не принято»)
  выполняет секретарь при составлении протокола;
- протокол загружается секретарём готовым файлом (не генерируется системой);
- поручения назначаются в первую очередь функциональным директорам,
  жизненный цикл: Новое → В работе → Исполнено → Принято секретарём
  (с возможностью возврата в работу).

## 3. Рабочий стол: календарь как центральный элемент

Рабочий стол построен вокруг месячного календаря — это первое, что видит
член СД при входе. В календаре в одной ленте сведены:

- заседания СД (дата проведения) — красным;
- сроки окончания приёма голосов по заочным заседаниям — жёлтым;
- сроки исполнения поручений — фиолетовым;
- рабочие созвоны, интервью и иные события, которые секретарь добавляет
  вручную прямо на нужный день, — синим/золотым/серым.

Клик по дню открывает панель справа со списком событий этого дня; клик по
заседанию или поручению в этой панели ведёт на соответствующую карточку.
Блоки «Требует вашего действия» и «Уведомления» оставлены под календарём —
они дополняют календарь, а не конкурируют с ним за внимание. У функционального
директора в календаре — только сроки его собственных поручений; заседания
и организационные события СД ему не видны (директор не входит в состав СД).

Технически: события собираются на лету (`GET /api/calendar`) из таблиц
`Meeting` и `Task`, плюс отдельная таблица `Event` — только для событий,
которые секретарь вносит вручную. Дублирования данных нет: если, например,
дата заседания в реестре меняется, она автоматически подтягивается и в
календарь.

## 4. Фирменный логотип

Логотип (файл `frontend/assets/logo.png`) размещён слева от надписи
MARSHALL — на экране входа и в шапке рабочей области. При замене логотипа

формат и примерно тот же размер).

## 5. Что уже реализовано

- Аутентификация: bcrypt-хэши паролей, JWT-сессии, блокировка учётной записи
  на 15 минут после 5 неудачных попыток входа.
- Заседания: создание с повесткой, статусы (планируется → голосование →
  завершено), согласование даты (опрос с вариантами).
- Голосование: за/против/воздержался, однократность, индикация кворума.
- Материалы: загрузка (только офисные форматы, до 25 МБ), автоверсионирование
  при повторной загрузке файла с тем же именем, скачивание с записью в журнал.
- Комментарии членов СД к вопросам повестки до заседания.
- Поручения: полный цикл контроля исполнения, отчёты исполнителей.
- Внутренние уведомления (создание заседания, старт голосования, новые
  материалы, назначение поручений, публикация протокола).
- Аудит-лог: только добавление записей, API изменения не предусматривает.

## 6. Доработка для переноса в закрытый контур (для ДЦТиИТ)

1. **База данных**: заменить `DATABASE_URL` на PostgreSQL
   (`postgresql+psycopg://…`), добавить `psycopg[binary]` в requirements,
   настроить регулярное резервное копирование БД и каталога `uploads/`.
2. **Аутентификация**: заменить `authenticate()` в `security.py` на проверку
   через корпоративный AD/LDAP или SSO (Kerberos/SAML). Остальной код от
   способа аутентификации не зависит. Задать `JWT_SECRET` переменной окружения.
3. **Файловое хранилище**: каталог `uploads/` заменить на корпоративное
   хранилище; добавить антивирусную проверку загружаемых файлов.
4. **Почтовые уведомления**: в `main.py` функция `notify()` — единая точка;
   добавить туда отправку через корпоративный SMTP/Exchange и планировщик
   напоминаний о сроках (голосование, поручения).
5. **ЭЦП**: в модели заложено место для расширения — при необходимости
   подписания протоколов/бюллетеней УКЭП интегрировать КриптоПро DSS или
   аналогичное решение на этапе загрузки протокола и подачи голоса.
   Юридическую схему (простая ЭП по соглашению сторон vs УКЭП) согласовать
   с юридическим департаментом до реализации.
6. **Фронтенд**: перевести на сборку Vite + TypeScript, библиотеки — локально
   (без внешних CDN), включить Content-Security-Policy.
7. **Транспорт**: HTTPS с корпоративным сертификатом (reverse proxy —
   nginx/traefik), HSTS.
8. **Эксплуатация**: логирование в корпоративную систему (SIEM), мониторинг,
   ограничение частоты запросов (rate limiting) на уровне прокси.

## 7. Известные ограничения MVP (не использовать в продакшне без устранения)

- Пароли демо-пользователей одинаковы и известны — сменить немедленно.
- JWT-секрет по умолчанию зашит в коде для dev-режима — задать через окружение.
- Нет HTTPS (MVP запускается локально) — обязателен в контуре.
- Нет антивирусной проверки загружаемых файлов.
- Фронтенд использует внешние CDN (React, шрифты Google) — недопустимо
  в закрытом контуре, см. п. 4.6.
- Нет автоматического закрытия заочного голосования по наступлению срока —
  сейчас голосование закрывает секретарь вручную; при доработке добавить
  планировщик (APScheduler/cron).
- Управление пользователями (создание/блокировка) — только через БД/seed;
  интерфейс администрирования пользователей добавить при доработке
  (или он станет ненужным при переходе на AD).
- SQLite не рассчитан на параллельную нагрузку — для контура PostgreSQL.
- Календарь не поддерживает повторяющиеся события и перетаскивание
  (drag-and-drop) между днями — событие переносится удалением и повторным
  добавлением на новую дату.
- Даты заседаний и поручений хранятся в текстовом формате ДД.ММ.ГГГГ (так
  привычнее для протоколов); при переносе в контур стоит перейти на единый
  тип «дата» в БД, чтобы исключить ошибки разбора формата.
